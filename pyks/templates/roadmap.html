Big Picture

The current PKI for OpenPGP is mainly based on Key Servers (KS) distributed across the internet that synchronize their databasis with some protocols. Phil Pennock gives an excellent and quite up-to-date Overview about OpenPGP Key Servers. You OpenPGP client (C) connects to one of them using the Synchronized : Your client within your Local Area Network (LAN) contacts a Public Key Server (PKS) in the internet (Wide Area Network, WAN):

With PyKS I propose an extension to the existing Public Key Infrastructure (PKI) that incorporates some answers to current threats that are now realized by many users not just the "paranoid" ones. Therefore I divided the KS component into a local key server LKS and a remote key server RKS.

Roadmap

Step 1 - Lokal Key Server (LKS)

First I will offer a local HKP key server that can be used with GnuPG and Thunderbird/Enigmail. This could be used as a local key server for small teams or organizations (experimental!) that do not want to publish their keys due to some of the reasons mentioned in my motivation. Furthermore this part could be used to implement some heuristics to detect manipulated keys (see Ralf Sendereks work). Currently I'm working

Step 2 - Lokal to Remote Key Server

In the next step I will built a second server that will speak a flavour of HKP but does only store encrypted public key certificates and hides the real user identities and key fingerprints by hashing. That part will be implemented in Python/sqlite in the first place. After that I might also implement a PHP/mysql version. That would allow the deployment onto commonly configured webspace.

Step 3 - Synchronizing of Remote Key Servers

Finally there should be a mechanism to synchronize the databases distributed across the internet.
